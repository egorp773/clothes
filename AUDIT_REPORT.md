# Аудит готовности marketplace-приложения

Дата: 2026-07-10

## Вердикт

Проект является UI-прототипом Flutter с частичной интеграцией Supabase. Он не
готов к публикации в App Store и не реализует безопасную сделку, реальную
доставку, оплату, персональный поиск или поиск по фото.

Проверки выполнены статически по коду и SQL, а также без чтения данных через
настроенный публичный Supabase endpoint. `flutter analyze` проходит. Все
локальные тесты проходят после исправления устаревшего smoke-теста (4/4).
REST endpoint из `lib/core/supabase_config.dart` вернул `404` для `products`,
`profiles`, `profile_private_details`, `orders` и `message_threads`; схема в
репозитории не доступна этому настроенному проекту.

## P0 — блокеры релиза

1. **Backend недоступен.** URL Supabase в
   `lib/core/supabase_config.dart` не предоставляет ни одну из ключевых
   таблиц. Любая синхронизация каталога, профиля, чата или заказов будет
   завершаться ошибкой и тихо оставлять пользователя на локальном кеше.

2. **Безопасной сделки нет.** `createDeliveryOrder` в
   `lib/data/app_repository.dart` создаёт заказ прямо с устройства и помещает
   его в `SharedPreferences` ещё до ответа сервера. Нет платёжного намерения,
   холда/списания/возврата, подписи webhook, идемпотентности, журнала денег,
   антифрода, разрешения споров и атомарного резервирования товара.

3. **RLS позволяет участнику заказа менять критичные поля.** Политика
   `Order participants can update orders` в `supabase/schema.sql` позволяет
   покупателю и продавцу менять всю строку: статус, цену, трек-номер, адрес и
   данные получателя. Состояние сделки нельзя доверять клиенту.

4. **PII смешаны с заказом.** В `orders` хранятся полный адрес, имя,
   телефон и email получателя. Их нельзя обновлять и отдавать обеим сторонам
   без ограничений статуса. Нужна отдельная таблица данных доставки с
   минимальным раскрытием после успешной оплаты.

5. **Push-уведомления — заглушка.**
   `lib/services/push_notification_service.dart` всегда возвращает
   `isEnabled = false` и `null`-токен. Серверная функция отправки есть, но
   клиент никогда не регистрирует устройство.

6. **Поиск и поиск по фото не реализованы.** В
   `lib/screens/catalog_screen.dart` обе кнопки вызывают только Snackbar.
   Нет текстового индекса, фильтрации на сервере, обработки изображения,
   embeddings или vector search.

7. **Текущая политика storage небезопасна для marketplace.** Любой
   авторизованный пользователь может загружать/обновлять любой путь в
   `product-images`; владельца файла и допустимые типы/размеры политика не
   проверяет. У bucket публичное чтение.

8. **iOS-конфигурация не релизная.** Bundle ID и redirect scheme остаются
   `com.example.clothes`; APNs entitlement — `development`; отсутствуют
   описания причин доступа к фото/камере при использовании `image_picker`.
   Название приложения — `Clothes`.

## P1 — существенные риски качества и данных

- Таблица `profiles` содержит только публичную витрину. Личные поля вынесены
  в `profile_private_details`, что правильно, но отсутствуют статусы
  верификации, роли, доверенные устройства, согласия на обработку данных,
  возрастные ограничения, история блокировок и признаки антифрода.
- Рейтинг и число продаж обновляются приложением по всем отзывам. Любой
  покупатель может оставить отзыв без завершённой покупки; участник чата может
  переписать JSON-массив сообщений целиком.
- Состояния заказа существуют только как enum интерфейса. Нет проверяемого на
  сервере автомата переходов и аудита переходов.
- Каталог загружается целиком каждые несколько секунд. Нет пагинации,
  серверной фильтрации, полнотекстового поиска, лимитов и кэш-стратегии.
- «Рекомендации» — список последних 24 просмотренных ID; текущая сортировка
  `По рекомендациям` не меняет выдачу. Нет событий клика/скролла/покупки,
  профиля интересов, негативных сигналов, diversity или метрик качества.
- В проекте остаются демо-данные, локальные `local_` заказы, mock-изображения,
  экран «Будет доступно потом» и Share Snackbar вместо системного шаринга.
- Нет миграционной истории: вся схема хранится одним идемпотентным
  `supabase/schema.sql`, что не позволяет надёжно воспроизводить production.

## P2 — эксплуатация и App Store

- README остаётся шаблоном Flutter; нет документации окружений, деплоя,
  отката, backup/restore и runbook.
- В репозитории нет CI, e2e, интеграционных, нагрузочных, security или
  контрактных тестов. До исправления smoke-тест был неактуален.
- Нет crash reporting, продуктовой аналитики, мониторинга backend, очередей и
  алертов.
- Для App Store ещё нужны финальные название/иконки/скриншоты, privacy policy,
  Privacy Manifest/labels, production APNs, удаление аккаунта (код уже
  намечен), moderation/report/block flow и Sign in with Apple при сохранении
  сторонних способов входа.

## Целевая архитектура

1. **Транзакции:** backend Edge Functions + Postgres RPC создают checkout
   атомарно и блокируют объявление. Платёжный провайдер marketplace-класса
   создаёт payment intent; только подписанный webhook меняет состояние.
   Нужны `payments`, неизменяемый double-entry `ledger`, `order_events`,
   `disputes`, `refunds` и таблицы доставки/отправлений.
2. **Доставка:** адаптер перевозчика создаёт отправление на сервере, хранит
   внешний ID, получает status webhooks и не принимает трек-номер из клиента.
3. **Рекомендации:** append-only `interaction_events` (view, dwell, search,
   click, favourite, cart, purchase, hide), item-атрибуты (категория, бренд,
   пол, размер, палитра, цена) и серверный ranker. Для джинсов вес категории,
   цвета, силуэта, бренда и ценового диапазона растёт по свежим сигналам, а
   уже купленное и скрытое исключается.
4. **Поиск по фото:** обработка загруженного файла на сервере, модерация и
   извлечение CLIP/SigLIP embedding; `pgvector`/векторный сервис для nearest
   neighbours; гибридный ранкер объединяет embedding, текст, фильтры и
   доступность товара. Никаких ключей ML/платежей в Flutter-клиенте.

## Порядок исправления

1. Восстановить/создать отдельные development и production Supabase-проекты,
   применить версионируемые миграции, удалить локальный fallback для денег и
   заказов.
2. Выбрать платёжного marketplace-провайдера и перевозчиков, оформить
   договоры/KYC/AML и секреты. Без этого нельзя честно включить «безопасную
   сделку».
3. Реализовать server-authoritative checkout, webhooks, dispute flow и
   строгие RLS/Storage-политики; покрыть их интеграционными тестами.
4. Заменить кнопки поиска, фото-поиска, push и share на реальные функции.
5. Построить event pipeline, рекомендации и image retrieval, затем проверить
   качество на анонимизированных данных и A/B-метриках.
6. Завершить iOS release-конфигурацию и пройти QA на реальных устройствах.

Пока не закрыт хотя бы каждый пункт P0, выпуск в App Store и приём реальных
платежей запрещены.
