# Удаление и обезличивание аккаунта

## Пользовательское обещание

Приложение не обещает «стереть абсолютно всё». После подтвержденного запроса:

- удаляются или необратимо обезличиваются профиль, delivery defaults, push
  tokens, активные объявления и иные ПДн, которые больше не нужны;
- публичный контент скрывается до обработки;
- необходимые записи сделки, оплаты/выплаты/возврата, спора, согласия,
  модерации и безопасности сохраняются только при наличии цели и основания,
  с ограниченным доступом и сроком из утвержденного retention schedule;
- сохраненные записи больше не должны позволять обычному пользователю или
  продавцу определить удаленного участника.

Оператор обязан опубликовать конкретные категории, основания и сроки до
production. Этот документ задает технический контракт, но не придумывает срок
за владельца данных.

## State machine запроса

```text
requested -> blocked_active_transaction
          -> approved -> anonymizing -> anonymized -> auth_deleted
          -> failed_retryable
          -> rejected_legal_hold
```

Запрос идемпотентен. Повтор с тем же ключом возвращает существующий результат,
а не запускает второе удаление.

## Что блокирует немедленную обработку

- заказ в любом незавершенном состоянии;
- открытый спор, возврат, chargeback или невыпущенная выплата;
- обязательный legal/security hold;
- незавершенная финансовая сверка или расследование fraud.

Блокировка удаления не восстанавливает доступ к продаже: аккаунт немедленно
переводится в ограниченный режим, новые публикации/заказы/сообщения запрещаются.
Пользователь получает request id, статус и понятную причину отсрочки.

## Server-side workflow

1. Edge Function `delete-account` проверяет Bearer JWT, недавнюю повторную
   аутентификацию и `Idempotency-Key`.
2. Сервер в транзакции создает `account_deletion_requests`, блокирует новые
   юридически значимые действия и проверяет active transaction/legal hold.
3. При допустимости server-only RPC:
   - скрывает и архивирует объявления;
   - отзывает seller access и push tokens;
   - удаляет delivery/profile PII;
   - заменяет публичные snapshots нейтральным псевдонимом;
   - отвязывает retained orders/disputes/events/audit от auth identity либо
     сохраняет внутренний псевдоним;
   - создает tombstones для необходимой истории чата, не уничтожая
     доказательства другого участника;
   - формирует список действительно удаляемых Storage objects.
4. Edge удаляет только возвращенный server-side список media. Attachment под
   dispute/legal hold не удаляется.
5. После фиксации `anonymized` удаляется Auth identity и отзываются provider
   tokens. Ошибка внешнего шага переводит запрос в `failed_retryable`; уже
   обезличенные данные не восстанавливаются.
6. Ответ содержит `request_id`, `status`, `anonymized_at`,
   `retained_categories` и при отсрочке `blocking_reasons`.

Нельзя полагаться на `ON DELETE CASCADE` для orders, order events, disputes,
payment records, audit log, reports или evidence. Их внешние ключи должны
поддерживать псевдонимизацию/`SET NULL`, а доступ — server-side retention rules.

## Клиент

Клиент работает fail-closed:

- при сетевой ошибке или неизвестном ответе показывает status запроса и не
  заявляет об успешном удалении;
- после `anonymized`/`auth_deleted` очищает session, profile, delivery, order и
  chat caches, media thumbnails и scoped preferences;
- для `blocked_active_transaction` завершает обычный доступ, но оставляет
  минимальный read-only экран статуса/спора/поддержки;
- не удаляет server attachments самостоятельно и не использует локальный кэш
  как подтверждение результата.

## Retention и контроль

До production владелец назначает срок и основание для каждой категории:

| Категория | Обработка после запроса |
|---|---|
| Публичный профиль и delivery defaults | удалить/обезличить |
| Draft и непроданные объявления | скрыть, затем удалить по schedule |
| Завершенные заказы и финансовые события | псевдонимизировать и сохранить ограниченный срок |
| Открытые сделки/споры | legal hold до завершения, затем schedule |
| Согласия и версии документов | сохранить доказательство в минимальном составе |
| Reports/moderator/security audit | псевдонимизировать, ограничить доступ |
| Chat без связи со спором | tombstone/удаление по schedule |
| Chat/evidence сделки или спора | legal hold и ограниченный доступ |
| Backups | удалить ротацией, запретить обычное восстановление учетной записи |

Нужны метрики pending/blocked/failed requests, alert на превышение SLA, журнал
оператора, регулярная проверка backup expiry и тест восстановления, который не
«воскрешает» удаленный аккаунт.
